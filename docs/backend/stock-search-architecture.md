# MaeMoJi 토스 스타일 종목검색 아키텍처

## 목표

MaeMoJi의 종목검색은 토스나 일반 증권앱처럼 `빠르게`, `깔끔하게`, `안정적으로` 동작해야 합니다.

핵심 목표는 아래와 같습니다.

- 검색 입력 시 즉시 결과가 떠야 함
- 외부 API 지연에 검색 UX가 영향을 받지 않아야 함
- 한글명, 영문명, 티커, 로고를 일관되게 보여줘야 함
- 포트폴리오 등록과 추천 엔진이 같은 종목 마스터를 바라봐야 함

## 결론

종목검색은 `외부 API 실시간 호출`이 아니라 `우리 DB 검색`이 맞습니다.

즉 구조는 이렇게 갑니다.

1. 사용자는 앱에서 종목 검색
2. WAS는 `stocks` 테이블에서 바로 검색
3. `stocks` 테이블은 배치가 외부 데이터 소스에서 주기적으로 갱신
4. 앱은 빠른 검색 결과만 받고, 외부 API 지연은 느끼지 않음

## 추천 구조

### 1. 앱

Flutter 앱은 검색어를 입력하면 우리 WAS의 검색 API만 호출합니다.

- 예: `GET /api/stocks/search?keyword=애플`
- 결과: 종목명, 티커, 거래소, 로고

앱은 절대 Finnhub 같은 외부 API를 직접 호출하지 않습니다.

## 2. WAS

WAS는 검색 요청을 받으면 `stocks` 테이블만 조회합니다.

역할:

- 검색어 정규화
- 한글명/영문명/티커 검색
- 정렬 우선순위 적용
- 결과 개수 제한

예시 정렬 우선순위:

1. 티커 완전일치
2. 한글명 완전일치
3. 영문명 완전일치
4. 티커 prefix 일치
5. 이름 prefix 일치
6. 나머지 부분일치

## 3. DB

검색의 핵심은 `stocks` 마스터 테이블입니다.

최소 컬럼 추천:

- `id`
- `ticker`
- `exchange_code`
- `finnhub_symbol`
- `name_ko`
- `name_en`
- `logo_url`
- `market_type`
- `is_active`
- `created_at`
- `updated_at`

검색 최적화용으로 추후 아래 확장도 좋습니다.

- `search_text`
- `name_ko_normalized`
- `name_en_normalized`
- `ticker_normalized`

## 4. 배치

배치는 검색용 종목 마스터를 채우고 갱신하는 역할입니다.

중요한 점:

- 검색 시 배치가 도는 게 아님
- 배치는 백오피스/운영 작업
- 검색은 항상 DB만 봄

### 배치 역할

1. 신규 종목 추가
2. 회사명 변경 반영
3. 거래소 변경 반영
4. 로고 갱신
5. 비활성 종목 정리

## 어떤 데이터를 언제 갱신하나

### 종목 기본정보

- 회사명
- 티커
- 거래소
- 로고
- 종목 활성 여부

이건 하루 1회 배치로도 충분한 경우가 많습니다.

예:

- 매일 새벽 1시
- 또는 미국 장 마감 후

### 시세 데이터

- 현재가
- 등락률
- 시가총액

이건 종목검색용 기본정보와 분리해서 생각하는 게 좋습니다.

검색 결과에 꼭 필요하지 않다면 `stocks`에 매번 넣지 않아도 됩니다.

즉:

- 검색용 정보는 `stocks`
- 시세 정보는 별도 캐시/스냅샷/실시간 API

## 실제 서비스 흐름

### 검색 흐름

1. 사용자가 `애플` 입력
2. Flutter가 WAS 검색 API 호출
3. WAS가 `stocks`에서 검색
4. DB 결과를 바로 반환

### 등록 흐름

1. 사용자가 검색 결과에서 종목 선택
2. `stock_id` 기준으로 포트폴리오 등록
3. 이후 추천 엔진도 같은 `stock_id` 사용

## 왜 이 구조가 좋은가

- 검색 속도가 빠름
- API 장애가 검색 UX에 직접 영향을 덜 줌
- 비용 예측이 쉬움
- 로고/한글명 품질을 우리가 통제 가능
- 추천 엔진과 데이터 연결이 안정적

## MaeMoJi 기준 권장 아키텍처

현재 기준으로 가장 적절한 구조는 아래입니다.

1. `stocks`를 검색 마스터 테이블로 사용
2. 앱 검색은 `GET /api/stocks/search`만 호출
3. 외부 API는 배치에서만 사용
4. 배치는 하루 1회 기본정보 갱신
5. 필요 시 장중 시세 배치는 별도로 분리

## 다음 구현 순서 추천

1. `stocks`를 종목검색 마스터로 확정
2. 검색 API를 `stocks` 전용으로 고도화
3. 배치 테이블/로그 구조 설계
4. 종목 마스터 적재용 배치 구현
5. 시세 갱신 배치는 별도 2차 작업으로 분리

## 한 줄 결론

토스 스타일 종목검색을 만들고 싶다면:

`검색은 DB에서 바로, 외부 API는 배치에서만`

이 방향이 맞습니다.
